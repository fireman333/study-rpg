## MODIFIED Requirements

### Requirement: New save SHALL seed 2 P5 starter doctors covering ward and surgery

`ensureSeed` SHALL insert two starter doctors into the `doctors` IndexedDB table when a fresh save is created (i.e., when the table is empty AND `gameCounters.hasUsedStarterPull === false`):

- 內科 P5 starter: `{ subjectId: "內科", rarity: "P5", powerMultiplier: 0.5, name: "內科 醫師 #1", spriteKey: "doctor-內科-P5", obtainedAt: Date.now(), assignedRoom: null }`
- 外科 P5 starter: `{ subjectId: "外科", rarity: "P5", powerMultiplier: 0.5, name: "外科 醫師 #1", spriteKey: "doctor-外科-P5", obtainedAt: Date.now() + 1, assignedRoom: null }`

The `id` field SHALL be generated by `crypto.randomUUID()` (or fallback unique id mechanism consistent with `recruitment-gacha` spec). The two starter doctors SHALL cover `ward` (內科) and `surgery` (外科) room types per the `SUBJECT_TO_ROOM` mapping in `affinity.ts`, ensuring the player can immediately assign at least one doctor to each of these room types in 診所 tier.

Seeding SHALL be idempotent for the **non-empty case** — if the `doctors` table already contains records (e.g., dogfood saves with existing doctors), seeding SHALL be skipped entirely. However, for **migrated v3 → v4 saves where the `doctors` table is empty** (regardless of `hasUsedStarterPull` value), `ensureSeed` SHALL execute a recovery branch: seed the 2 P5 starter doctors AND reset `gameCounters.hasUsedStarterPull` to `false` so the `StarterPullCard` becomes visible. This recovery branch handles two sub-cases atomically:

1. `hasUsedStarterPull === undefined` (raw v3 migration, never patched) — seed + set flag to `false`.
2. `hasUsedStarterPull === true` (v3 migration already patched by prior buggy boot) — seed + reset flag to `false`.

In both sub-cases, the recovery branch is self-terminating: after firing once, `doctorCount > 0` and subsequent boots skip the recovery codepath.

#### Scenario: Fresh save receives 2 starter doctors

- **GIVEN** a fresh `apps/medexam2-hospital-tw` save (no IndexedDB records)
- **WHEN** `ensureSeed` runs for the first time
- **THEN** the `doctors` table SHALL contain exactly 2 records
- **AND** one record SHALL have `subjectId = "內科"`, `rarity = "P5"`, `powerMultiplier = 0.5`, `name = "內科 醫師 #1"`
- **AND** the other record SHALL have `subjectId = "外科"`, `rarity = "P5"`, `powerMultiplier = 0.5`, `name = "外科 醫師 #1"`
- **AND** both records SHALL have `assignedRoom = null`
- **AND** `gameCounters.hasUsedStarterPull` SHALL be `false`

#### Scenario: Existing save with doctors unchanged by seed run

- **GIVEN** an existing save with 5 doctors in the `doctors` table
- **WHEN** `ensureSeed` runs at app boot
- **THEN** the `doctors` table SHALL remain at 5 records
- **AND** no new starter doctors SHALL be inserted

#### Scenario: v3 → v4 migrated save WITH existing doctors preserves flag

- **GIVEN** a dogfood v3 save with 3 doctors and `gameCounters.hasUsedStarterPull === undefined` (v3 schema did not have this field)
- **WHEN** `ensureSeed` runs after Dexie v4 upgrade completes
- **THEN** `gameCounters.hasUsedStarterPull` SHALL be set to `true`
- **AND** the `doctors` table SHALL remain at 3 records (no starter doctors added)
- **AND** the StarterPull UI affordance SHALL NOT appear for this save

#### Scenario: v3 → v4 migrated save with EMPTY doctors recovers via seed

- **GIVEN** a dogfood v3 save with 0 doctors and `gameCounters.hasUsedStarterPull === undefined`
- **WHEN** `ensureSeed` runs after Dexie v4 upgrade completes
- **THEN** the recovery branch SHALL execute atomically inside the `ensureSeed` transaction
- **AND** the `doctors` table SHALL contain exactly 2 records (內科 P5 + 外科 P5 starter, generated by `makeStarterDoctor`)
- **AND** `gameCounters.hasUsedStarterPull` SHALL be `false`
- **AND** the `StarterPullCard` UI affordance SHALL be visible on next render

#### Scenario: Already-buggy save (flag=true + 0 doctors) self-heals on next boot

- **GIVEN** a save that previously hit the buggy v3 → v4 migration codepath, resulting in `gameCounters.hasUsedStarterPull === true` AND `doctors.count() === 0`
- **WHEN** the user loads the app with the patched build for the first time
- **AND** `ensureSeed` runs
- **THEN** the recovery branch SHALL detect `doctorCount === 0` regardless of flag value
- **AND** the `doctors` table SHALL contain exactly 2 records (內科 + 外科 P5 starters)
- **AND** `gameCounters.hasUsedStarterPull` SHALL be reset to `false`
- **AND** the `StarterPullCard` SHALL re-appear in the HomePage UI (Dexie liveQuery re-emits)

### Requirement: hasUsedStarterPull flag SHALL persist across the gameCounters table

The `gameCounters` table singleton row SHALL include a `hasUsedStarterPull: boolean` field. The field SHALL be set as follows:

- **Fresh v4 save**: `hasUsedStarterPull = false`
- **Migrated v3 → v4 save with existing doctors**: `hasUsedStarterPull = true` (preserve original intent — prevent dogfood users from being re-prompted)
- **Migrated v3 → v4 save with 0 doctors**: `hasUsedStarterPull = false` (recovery — see Requirement 1's recovery branch)
- **After successful starter pull**: `hasUsedStarterPull = true`
- **In normal play, once true, never reset**: There SHALL be no UI affordance, gameplay action, or migration path that resets this flag to `false` after the player has used their starter pull. The ONLY codepath permitted to set `hasUsedStarterPull` from `true` (or `undefined`) back to `false` is the `ensureSeed` recovery branch when it detects `doctorCount === 0` on a save that has a `gameCounters` row.

The Dexie schema migration to v4 SHALL include a hook (in `ensureSeed`, NOT in the Dexie `.upgrade()` callback) that patches existing `gameCounters` rows based on the doctor-count branch described in Requirement 1.

#### Scenario: Fresh v4 save has flag false

- **GIVEN** a fresh v4 save
- **WHEN** `gameCounters` is read after `ensureSeed`
- **THEN** the singleton row SHALL have `hasUsedStarterPull = false`

#### Scenario: Migrated v3 → v4 save WITH existing doctors has flag true

- **GIVEN** a v3 save with 3 doctors and `gameCounters` row that does not contain `hasUsedStarterPull` field
- **WHEN** Dexie upgrades to v4 and `ensureSeed` runs
- **THEN** the `gameCounters` row SHALL have `hasUsedStarterPull = true`

#### Scenario: Migrated v3 → v4 save WITH 0 doctors has flag false

- **GIVEN** a v3 save with 0 doctors and `gameCounters` row that does not contain `hasUsedStarterPull` field
- **WHEN** Dexie upgrades to v4 and `ensureSeed` runs
- **THEN** the `gameCounters` row SHALL have `hasUsedStarterPull = false`
- **AND** the `doctors` table SHALL be seeded with 2 P5 starters in the same transaction

#### Scenario: Flag becomes true after starter pull

- **GIVEN** `gameCounters.hasUsedStarterPull = false`
- **WHEN** the player completes a starter pull
- **THEN** the same row SHALL have `hasUsedStarterPull = true` after the operation completes
- **AND** the value SHALL persist across app reload (Dexie write to disk)

#### Scenario: Flag value never resets in normal play (only recovery branch may reset it)

- **GIVEN** `gameCounters.hasUsedStarterPull = true` AND `doctors.count() > 0`
- **WHEN** the player performs any normal action (recruit doctor, answer quiz, assign room, upgrade tier, refresh page, etc.)
- **THEN** `hasUsedStarterPull` SHALL remain `true`
- **AND** no codepath OTHER THAN `ensureSeed`'s recovery branch SHALL set the flag back to `false`
- **AND** `ensureSeed`'s recovery branch SHALL NOT fire when `doctors.count() > 0`
